# wiki ↔ repo 동기화 프로토콜

wiki와 코드 repo가 **어긋나지 않게 유지하는 운영 규약**. 왜 이렇게 정했는지(대안 기각 포함)는 결정 기록 [[0001-repo-wiki-operating-model]], 이 페이지는 **어떻게 돌아가는지**의 살아있는 레퍼런스.

한 줄: **코드의 진실은 git, 의도·지식의 진실은 wiki. 의존은 한 방향(repo ⟸ wiki) — wiki가 repo를 읽어 흡수하되, repo는 wiki를 모른다.**

---

## 왜 드리프트가 생겼나 (해소 대상)

종전엔 작업 현황·지식을 **오직 "환원 프롬프트" 서술로만** wiki에 갱신했다. 그래서:

1. 환원이 lossy — 요약이 실제 커밋을 다 반영 못 함.
2. 환원 누락 — 세션을 환원 없이 끝내면 그 작업분을 wiki가 영영 모름.
3. wiki 밖 작업 — 사이클 안 태운 커밋.
4. 시차 — 갱신 시점과 작업 시점의 gap.

근본 원인: **"뭐가 구현됐나"의 진실은 git에 있는데 wiki가 그걸 서술로 복제**해서 필연적으로 어긋났다.

---

## 두 채널

| 종류 | 진실의 원천 | 넘기는 법 |
|------|------------|-----------|
| **작업 현황** | 코드 repo의 git 히스토리 | wiki가 `git log <앵커>..HEAD`를 **직접 읽어** 파악 |
| **경험·지식** | repo 안 약속된 경로(`docs/log.md`, `docs/<topic>.md`) | 코드 세션이 거기 쌓고, wiki가 **직접 읽음** |

복붙(환원 프롬프트 수동 전달)이 사라지고 그 자리를 **"repo가 기억하고 wiki가 읽는다"**가 대신한다.

---

## 앵커 — 흡수 경계선

앵커 = 각 프로젝트 `status.md`에 적어두는 **git 커밋 SHA 마커**. 의미는 "wiki는 이 프로젝트를 **이 커밋까지 흡수 완료**했다".

- 다음 동기화 때 읽을 델타 = `git log <앵커>..HEAD`.
- 흡수를 마치면 앵커를 HEAD(또는 새 main tip)로 **전진**시킨다.
- **브랜치 지형**: 작업이 feature 브랜치에서 나서 main에 squash 병합되는 흐름이면, 앵커는 **트렁크(main) tip**을 durable 기준으로 두고 **진행 중 브랜치 tip**을 별도 표기한다. 브랜치가 병합되면(새 SHA 생성) main 앵커를 새 tip으로 전진, 진행-브랜치 앵커는 리셋. (선례: 8kw [[status]] 앵커 블록.)

앵커가 하는 일은 **드레인 표시 + 델타 리더 겸용** — 이 하나로 싱크·중복·토큰을 같이 잡는다.

---

## 토큰 비용 — 파일 크기와 무관

wiki는 `docs/log.md` **전체를 읽지 않는다.** 앵커가 있으니 델타만 읽는다:

```bash
git log <앵커>..HEAD --stat -- docs/    # docs가 바뀌긴 했나 먼저 (안 바뀌었으면 읽기 0)
git diff <앵커>..HEAD -- docs/log.md docs/   # 앵커 이후 추가된 줄만
```

→ 읽기 비용 = **마지막 동기화 이후 작업량**에 비례. log.md가 10줄이든 만 줄이든 무관. "매번 docs 전체를 읽어 토큰 폭발"은 일어나지 않는다.

그래서 **log.md 길이는 사람 가독성 문제일 뿐** — 거슬리면 분기 로테이션(`docs/log/YYYY-QN.md`로 이동, git이 히스토리 보존). 로테이션한 세션엔 wiki에 한마디 해주면 diff 소음을 감안해 읽는다.

---

## 코드 세션은 wiki를 모른다 (한 방향 의존)

의존은 `repo ⟸ wiki` 한 방향이어야 깨끗하다. 코드 세션이 "wiki에 넣을 것"을 의식하면 화살표가 거꾸로 생겨 repo에 바깥 wiki 때문에만 존재하는 인공물이 낀다(누수).

그래서 repo 쪽 규약은 **wiki를 언급하지 않는 보편적 엔지니어링 위생**으로 표현한다:

> 중요한 결정·함정·시행착오·측정은 `docs/log.md`에 날짜별로 남긴다. 한 주제가 깊어지면 `docs/<topic>.md`로 분리한다.

- 이 지침의 집은 **`eta-harness` 플러그인의 coder(+verifier) 에이전트** — 코딩 워크플로 상시 활성이라 안정적으로 발동하고, 모든 펌웨어 프로젝트가 상속(DRY). 프로젝트별 CLAUDE.md에 복붙하지 않는다.
- **lazy-create**: `docs/`가 없으면 첫 로그 쓸 때 만든다(`mkdir -p`). 미리 프로비저닝 안 함 — 빈 `docs/`가 안 생긴다.
- 로그·토픽노트는 **wiki가 없어도 가치 있는** repo 자산(다음 엔지니어·리뷰·미래의 나를 위해).

wiki(읽는 쪽)는 `docs/` 부재를 **에러 없이 degrade**: git으로 작업 현황은 그대로 흡수, 지식은 "아직 로그 채널 없음"으로 넘어간다. → 프로젝트 성숙도가 달라도 안 깨진다.

**두 세계를 잇는 유일한 존재는 사람(오케스트레이터)** — "동기화해줘" 한마디가 다리. 내용을 나르는 게 아니라 스위치를 누르는 것.

---

## 무엇을 지식으로 담나

git이 이미 아는 건 담지 않는다(무엇을 했다·diff 내용). 코드를 봐도 안 나오는 것만:

- **왜** 그 결정을 했나 (버린 대안 포함)
- **놀란 점·함정** — "데이터시트 X vs 실측 Y"
- **막다른 길** — 시도했다 안 된 것 + 이유 (재시도 방지)
- **매직넘버의 출처** — N=16이 왜 16인지

---

## 결정(decision) 흐름

결정은 **의도**라 진실의 집이 wiki다. 두 갈래로 들어와 모두 wiki `decisions/`(ADR)에 산다:

| 출처 | 흐름 |
|------|------|
| bottom-up | 코딩 중 나온 결정 → `docs/log.md` → wiki가 무게 있는 걸 ADR로 승격 |
| top-down | wiki에서 숙고해 내림 → 바로 ADR 작성 (roadmap·크로스-프로젝트 맥락 위에서) |

- 위치: 프로젝트 결정 = `teams/<팀>/<프로젝트>/decisions/NNNN-*.md`, 크로스-커팅/전사 결정 = 루트 `pages/decisions/NNNN-*.md`.
- 빈 `decisions/`는 미리 안 만든다(lazy) — 실제 결정이 생길 때 판다.
- **ADR 규율: 결정은 수정하지 않고 supersede.** 생각이 바뀌면 옛 레코드를 `status: superseded-by NNNN`으로 두고 새 레코드를 쓴다 — 생각이 바뀐 역사가 남는 게 결정 기록의 가치.
- 결정은 박제가 아니라 방향타: `status.md` "다음 시작점"과 `roadmap.md` 마일스톤이 ADR로 백링크 위임.

## 전체 그림

```
repo (진실: 코드)                wiki (의도·전략·지식)
─────────────────                ──────────────────────
git 커밋        ──현황(델타)──→   status.md (앵커로 델타만 읽음)
docs/log.md     ──재료──────────→ decisions/ (ADR)  ←숙고해 직접 작성도
docs/<topic>.md ──소스──────────→ concepts/ entities/ sources/
                                  roadmap.md ←결정이 방향 조정
                      ←─dispatch── "다음 작업" 프롬프트
```
