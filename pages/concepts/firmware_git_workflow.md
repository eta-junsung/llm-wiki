# 펌웨어 Git 워크플로 표준 (트렁크 기반 + 릴리스 태그)

모든 팀·프로젝트 공통. eta 펌웨어 팀이 앞으로 **모든 펌웨어 저장소**에 적용하는 git 브랜칭·릴리스 컨벤션. 단일 프로젝트의 결정이 아니라 팀 공통 규약이다.

> **한 줄 정의: 태그는 소스를 고정하고, 동일 바이너리는 고정된 툴체인+의존성과 함께일 때 보장된다.**

---

## 0. 현재 채택 상태 — Tier 계층 (지금 적용되는 것)

이 표준은 "팀 + 양산 + 멀티버전"을 가정한 **완성형**이다. 혼자 개발하는 현 단계에서 전부를 켜면 지키지 못해 문서와 실제가 따로 논다. 그래서 **"지금 지킬 최소(Tier 1) + 트리거 시 켤 스위치(Tier 2·3)"** 로 계층을 나눈다. 선제 도입은 금물.

| Tier | 트리거 | 항목 |
|------|--------|------|
| **Tier 1 — 지금 active (혼자여도)** | — | PR 필수(§3) · Conventional Commits(§4) · annotated 태그 + git hash 임베드(§5·§9) · 최소 CI 빌드 게이트(§8) · repo 루트 `CONTRIBUTING.md` |
| **Tier 2 — 두 번째 사람 / 첫 정식 릴리스** | 협업 시작 or `v1.0.0` | main 브랜치 보호 + PR 리뷰 require · CI에 정적분석/테스트 추가 · 코드 서명(§9) |
| **Tier 3 — 필드에 여러 버전 동시 생존** | 다중 SKU/고객 독립 패치 | `release/x.y` 유지 브랜치(§11) · 툴체인 Docker/Nix 핀 + reproducible build(§2.1) · west/매니페스트(§11) |

> **혼자여도 PR을 기본값으로 둔다** — 사람이 오면 "리뷰 require" 한 스위치만 켜면 0 변경으로 슬롯인된다. 습관이 시스템에 박혀 있어야 새 사람이 그대로 따른다.
>
> **코드 옆에도 둔다**: 각 repo 루트에 `CONTRIBUTING.md` 한 장으로 이 규칙을 요약하고 이 페이지를 링크한다. 새 사람이 wiki를 몰라도 repo만 보고 따를 수 있게.

## 1. 채택 모델 — 트렁크 기반 개발(TBD) + 릴리스 태그

- 중심 줄기(trunk)는 단 하나: `main`. 모든 개발이 여기로 모인다.
- Git Flow의 평행 줄기(`develop`, `release/*`)는 **기본으로 두지 않는다** — 소규모 단일 진화 제품에는 과하다. (동시 유지 버전이 여러 개가 되면 → §6·§11)
- 릴리스는 "브랜치"가 아니라 **annotated 태그**로 박는다. 태그는 특정 커밋에 영구 고정되는 이름표 — 보드에 구운 바로 그 버전을 되짚는 앵커.

## 2. 왜 펌웨어에 태그가 필수인가 (웹과 다른 점)

- **재현성**: 현장 보드에 문제가 나면 `git checkout <tag>`로 그 시점 소스를 정확히 되짚을 수 있어야 한다.
- **멀티-MCU 시스템**: 한 시스템이 여러 MCU 펌웨어로 구성될 때(예 01/02/03 — [[app_protocol_module]] 3펌웨어 구조), 태그 하나가 전체 MCU 이미지를 "한 세트의 시스템 스냅샷"으로 묶어 잡는다.
- 즉 태그는 선택이 아니라 **배포의 정의 그 자체**다.

### 2.1. 중요 — 태그는 *소스*만 고정한다. 동일 바이너리는 별도 조건

`git checkout <tag>`만으로 동일 바이너리가 나오지는 **않는다**. git 태그가 고정하는 것은 소스 트리뿐이고, 펌웨어 바이너리를 결정하는 입력은 훨씬 넓다:

- 툴체인 버전 (`gcc-arm-none-eabi` 10.3 vs 12.2면 바이너리가 달라짐 — cf. [[sdk_ccs_toolchain_migration]]: SDK/CGT 버전이 빌드 산출물을 바꾸는 실증)
- 벤더 SDK/HAL 버전 (ESP-IDF, STM32Cube, nRF SDK, Zephyr, TI MCU+ SDK 등)
- 빌드 플래그·최적화 레벨
- 의존 라이브러리 버전
- 빌드 타임스탬프 등 비결정적 요소 (`__DATE__` 임베드 등)

**따라서 "동일 바이너리"는 태그 + 고정된 빌드 환경이 함께일 때만 보장된다.** 반드시 같이 박는다 (→ Tier 3):

- 툴체인을 **Docker 이미지(또는 Nix)**로 고정하고 버전을 태그와 함께 기록
- 의존성 **락파일** (Zephyr `west.yml`, ESP-IDF `idf_component.lock`, PlatformIO 핀 등)
- 가능하면 reproducible build (`SOURCE_DATE_EPOCH`, 결정적 링킹)

## 3. 일상 흐름 — PR 필수(Tier 1)

| 상황 | 동작 |
|------|------|
| 모든 변경 | `feature/<주제>` 짧은 가지 → push → **PR** → CI green 확인 → main에 merge → 가지 삭제 |
| 배포 시점 | `git tag -a vX.Y.Z -m "..."` → `git push --tags` |
| 옛 배포본 복원 | `git checkout vX.Y.Z` |

- **혼자여도 PR을 거친다**(self-merge). 변경이 PR 단위로 묶이고 CI green이 박혀, 사람이 오면 "리뷰 require"만 켜면 끝. main 직접 push는 오타 수정 등 정말 사소한 것만 예외로 허용하되, 그것도 CI는 통과해야 한다.
- feature 가지는 짧게 살고 merge 후 삭제한다. 줄기 하나 + 잠깐 나는 가지 = 깔끔.
- **main 게이트**: "모든 타깃 MCU 빌드 green"을 merge 통과 조건으로 건다. 이 게이트를 기계로 강제하는 것이 §8 CI다. TBD는 CI를 생략한다는 뜻이 아니다 — 짧은 브랜치 + 잦은 통합 + 강한 CI다. 한 커밋이 전 MCU 빌드를 깨면 팀 전체가 멈춘다.

## 4. 커밋 메시지 — Conventional Commits

커밋 메시지는 미래의 나/남이 히스토리를 읽는 유일한 인터페이스다. **Conventional Commits** 규칙을 쓴다 — 기계가 파싱 가능해 나중에 자동 changelog·자동 버전범프로 이어진다.

형식: `<type>(<scope>): <한 줄 요약>` + (필요시 본문 + 푸터)

```
feat(pwm): dead-time knob 상한 500ns로 확장

기존 400ns로는 leg2 isoform 케이스를 못 덮음. 측정 근거는
pwm_deadtime_knob_verify 참조.

Refs: #12
```

| type | 언제 | SemVer 영향(§5) |
|------|------|------|
| `feat` | 기능 추가 | MINOR |
| `fix` | 버그 핫픽스 | PATCH |
| `build` | 빌드 시스템·툴체인·makefile | — |
| `refactor` | 동작 불변 구조 개선 | — |
| `docs` | 문서 | — |
| `test` | 테스트 | — |
| `chore` | 잡일(의존성 업뎃 등) | — |

- `scope`는 영역명(예 `pwm`, `flash`, `ble`, MCU면 `01`/`02`/`03`). 생략 가능.
- **호환 깨짐**은 `feat!:` 또는 본문에 `BREAKING CHANGE:` → MAJOR.
- 한 줄 요약은 명령형·현재형("추가한다"가 아니라 "추가"). 50자 내 권장, 본문은 "왜"를 적는다.

## 5. 버전 체계 — SemVer (vMAJOR.MINOR.PATCH)

- 정식 출시 전 단계는 `v0.x.y`로 시작 (예: 첫 중간 배포 = `v0.1.0`).
- **MINOR** (+0.1.0): 기능 추가/일반 배포.
- **PATCH** (+0.0.1): 핫픽스.
- **MAJOR**: 호환성 깨지는 큰 변경 (정식 출시 시 `v1.0.0`).
- 자리를 올리면 오른쪽 칸은 0으로 리셋 (`v0.4.2` + 기능 → `v0.5.0`).
- 태그는 반드시 **annotated (`-a`)** — 작성자/날짜/메시지가 남는다. **lightweight 금지.** `git describe`도 annotated만 깔끔히 잡는다.

## 6. 핫픽스 정책 — 기본 "앞으로만 진행", 다중 유지 시 전환

- 중간 배포는 임시본. 버그가 나면 트렁크에서 고치고 다음 태그로 배포한다.
- 유지(maintenance) 브랜치를 **미리** 만들지 않는다.
- **예외 1 (단발)**: 현장에 특정 배포 버전이 살아남아 그 버전만 독립 패치해야 하는 상황이 실제로 오면, 그때 해당 태그에서 `hotfix/*` 가지를 잠깐 쳐서 패치 → 패치 태그(`vX.Y.Z+1`) → 트렁크로 머지백. 미리 만들지 말고 필요할 때만.
- **예외 2 (지속) — 크로스오버 주의**: 펌웨어는 웹과 달리, 고객/양산에 나간 순간 그 버전만 독립 패치해야 할 확률이 높다. 여러 SKU/고객이 서로 다른 버전을 동시에 들면, 임시 `hotfix/*`보다 `release/x.y` **유지 브랜치 모델**(§11, Tier 3)로 넘어가는 것이 정답이다. "예외"가 "일상"이 되는 시점을 망설이지 말고 전환하라.

## 7. 브랜치 위생

- 정본은 항상 `main`. `origin/HEAD`가 main을 가리키게 유지(GitHub 기본·CI 관습).
- 병합 끝난 작업 브랜치는 로컬·원격 모두 삭제. 잔재 브랜치를 쌓지 않는다.
- 삭제 전 반드시 `git rev-list --count main..<branch>`로 트렁크에 없는 고유 커밋이 0인지 확인. 0이 아니면 버릴지/머지할지/아카이브 태그로 보존할지 판단.
- 버리긴 아까운 실험본(예: SDK 포팅 시도)은 `archive/<주제>` 태그로 tip을 박아 보존한 뒤 브랜치는 삭제 → 히스토리는 남고 브랜치 목록은 깔끔.

## 8. CI — 자동 빌드 게이트

**CI(Continuous Integration)**: push/PR마다 **깨끗한 서버에서** 빌드를 자동으로 돌려, 통과(green)해야만 merge를 허용하는 자동화. GitHub Actions 등이 수행한다.

### 8.1. 왜 — 로컬 빌드와 다른 점

로컬 빌드 성공은 내 PC의 툴체인 경로·환경변수·아직 커밋 안 한 파일 덕일 수 있다. CI는 **git에 올라간 것만으로** 빈 서버에서 빌드하므로:

1. **works-on-my-machine 차단** — 커밋 깜빡한 헤더, 내 PC에만 있는 경로를 잡는다. repo를 받은 다음 사람(새 PC·신입·미래의 나)이 진짜 빌드 가능한지 보증.
2. **약속 → 게이트** — "push 전 빌드"는 머릿속 약속이라 바쁘면 건너뛴다. CI는 기계가 강제 → 빌드 깨진 커밋은 main 진입 불가. §3의 main 게이트가 실재가 되는 지점.
3. **멀티-MCU 폭증 방지** — 01/02/03이 되면 "한 군데 고치고 나머지 빌드 깜빡"이 일상. CI는 항상 전부 빌드.

### 8.2. 전제 — 헤드리스(CLI) 빌드 가능해야 함

CI는 GUI 없는 리눅스 서버에서 명령어로 빌드한다. **IDE 클릭으로만 빌드되면 CI를 못 건다.** 프로젝트별로 갈린다:

| 프로젝트 (툴체인) | 헤드리스 빌드 | CI |
|---|---|---|
| 8kw-ev-wpt-tx · lp-am263p (TI AM263P, gmake) | `gmake -C build all` ✓ ([[build_methods]]) | **가능 — 지금** |
| nRF (SEGGER SES) | `emBuild` CLI [추정] | 가능[추정] |
| STM32CubeIDE (예: 01_RX_control) | CLI 빌드 **불가**, IDE Ctrl+B만 ([[cubeide_cli_build_trap]]) | 막힘 → CI 예외로 명시 |

[[linux_migration]]으로 개발 호스트가 Ubuntu로 가는 것과 정렬된다 — GitHub Actions 우분투 러너 = 개발환경. (전환 동기 ②의 "CI/CD·빌드팜"이 이 항목.)

### 8.3. 최소 시작 (Tier 1)

지금 거는 건 **빌드까지만**. PR 올라옴 → 우분투 러너가 툴체인 설치 → 빌드 스크립트(`gmake -C build all`) 실행 → 성공하면 green → merge 허용. 보드 실측(HIL)·정적분석·테스트는 Tier 2 이후.

## 9. 릴리스 추적성 (펌웨어 필수)

- **이미지에 git hash 임베드**: 현장 보드가 자기 버전(`git describe` 결과)을 보고할 수 있어야 태그 ↔ 펌웨어 매핑이 닫힌다. (Tier 1 항목)
- **바이너리 보관**: 태그는 `.bin`/`.hex`/`.elf`를 담지 못한다. CI 아티팩트나 릴리스 레지스트리에 보관하고, MCU별 `(커밋, 바이너리 해시)`를 **매니페스트**로 같이 박는다.
- **코드 서명**: secure boot를 쓰면 릴리스 태그 = 서명된 이미지. 릴리스 절차에 코드 서명 단계를 명시한다. (Tier 2)

## 10. 실무 주의

- 태그는 커밋된 상태에만 박힌다 — 배포 태그는 **워킹트리가 깨끗하고 push된 tip**에서.
- git 정리(브랜치 삭제·fast-forward·태그)는 되돌리기 까다로우니, 실행 전 명령 목록을 검토하고 push 전에 로컬에서 먼저 확인한다.

## 11. 참고 — 규모가 커질 때의 대안 모델 (Tier 3)

본 표준(TBD + 태그)은 **소규모 단일 진화 제품의 기본값**이다. 상황이 바뀌면 아래로 전환을 검토한다. 선제 도입은 금물.

| 모델 | 언제 |
|------|------|
| **GitHub Flow** (모든 변경 = 짧은 브랜치 + PR) | 본 표준 Tier 1이 이미 사실상 이것. 리뷰/CI 게이트를 명확히 하면 그대로 Tier 2. |
| **`release/x.y` 유지 브랜치** | 필드에 여러 버전이 동시에 살아 독립 패치가 필요해질 때. forward-only의 다음 단계. |
| **Git Flow** | 동시 유지 버전이 다수일 때. 단일 제품엔 과하지만 다중 유지엔 맞는 도구. |
| **매니페스트 / west 방식 (Zephyr)** | 멀티-MCU가 서로 다른 속도로 진화할 때. 매니페스트 저장소가 각 MCU 저장소의 SHA를 핀으로 박아 "한 세트 시스템 스냅샷"을 더 강하게 고정. MCU가 lockstep이면 모노레포+전체 태그로 충분. |
