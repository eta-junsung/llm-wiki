# 펌웨어 Git 워크플로 표준 (트렁크 기반 + 릴리스 태그)

모든 팀·프로젝트 공통. eta 펌웨어 팀이 앞으로 **모든 펌웨어 저장소**에 적용하는 git 브랜칭·릴리스 컨벤션. 단일 프로젝트의 결정이 아니라 팀 공통 규약이다.

> **한 줄 정의: 태그는 소스를 고정하고, 동일 바이너리는 고정된 툴체인+의존성과 함께일 때 보장된다.**

---

## 1. 채택 모델 — 트렁크 기반 개발(TBD) + 릴리스 태그

- 중심 줄기(trunk)는 단 하나: `main`. 모든 개발이 여기로 모인다.
- Git Flow의 평행 줄기(`develop`, `release/*`)는 **기본으로 두지 않는다** — 소규모 단일 진화 제품에는 과하다. (동시 유지 버전이 여러 개가 되면 → §5·§9)
- 릴리스는 "브랜치"가 아니라 **annotated 태그**로 박는다. 태그는 특정 커밋에 영구 고정되는 이름표 — 보드에 구운 바로 그 버전을 되짚는 앵커.

## 2. 왜 펌웨어에 태그가 필수인가 (웹과 다른 점)

- **재현성**: 현장 보드에 문제가 나면 `git checkout <tag>`로 그 시점 소스를 정확히 되짚을 수 있어야 한다.
- **멀티-MCU 시스템**: 한 시스템이 여러 MCU 펌웨어로 구성될 때(예 01/02/03 — [[app_protocol_module]] 3펌웨어 구조), 태그 하나가 전체 MCU 이미지를 "한 세트의 시스템 스냅샷"으로 묶어 잡는다.
- 즉 태그는 선택이 아니라 **배포의 정의 그 자체**다.

### 2.1. 중요 — 태그는 *소스*만 고정한다. 동일 바이너리는 별도 조건

`git checkout <tag>`만으로 동일 바이너리가 나오지는 **않는다**. git 태그가 고정하는 것은 소스 트리뿐이고, 펌웨어 바이너리를 결정하는 입력은 훨씬 넓다:

- 툴체인 버전 (`gcc-arm-none-eabi` 10.3 vs 12.2면 바이너리가 달라짐 — cf. [[sdk_ccs_toolchain_migration]]: SDK/CGT 버전이 빌드 산출물을 바꾸는 실증)
- 벤더 SDK/HAL 버전 (ESP-IDF, STM32Cube, nRF SDK, Zephyr, TI MCU+ SDK 등)
- 빌드 플래그·최적화 레벨
- 의존 라이브러리 버전
- 빌드 타임스탬프 등 비결정적 요소 (`__DATE__` 임베드 등)

**따라서 "동일 바이너리"는 태그 + 고정된 빌드 환경이 함께일 때만 보장된다.** 반드시 같이 박는다:

- 툴체인을 **Docker 이미지(또는 Nix)**로 고정하고 버전을 태그와 함께 기록
- 의존성 **락파일** (Zephyr `west.yml`, ESP-IDF `idf_component.lock`, PlatformIO 핀 등)
- 가능하면 reproducible build (`SOURCE_DATE_EPOCH`, 결정적 링킹)

## 3. 일상 흐름

| 상황 | 동작 |
|------|------|
| 작은 수정 | 짧은 브랜치 → CI green 확인 후 main에 merge (또는 PR) |
| 덩어리 작업 | `feature/<주제>` 가지 → 작업 → main에 merge → 가지 삭제 |
| 배포 시점 | `git tag -a vX.Y.Z -m "..."` → `git push --tags` |
| 옛 배포본 복원 | `git checkout vX.Y.Z` |

- feature 가지는 짧게 살고 merge 후 삭제한다. 줄기 하나 + 잠깐 나는 가지 = 깔끔.
- **main 게이트**: 직접 push라도 "모든 타깃 MCU 빌드 green"을 통과 조건으로 건다. TBD는 CI를 생략한다는 뜻이 아니다 — 짧은 브랜치 + 잦은 통합 + 강한 CI다. 한 커밋이 전 MCU 빌드를 깨면 팀 전체가 멈춘다.

## 4. 버전 체계 — SemVer (vMAJOR.MINOR.PATCH)

- 정식 출시 전 단계는 `v0.x.y`로 시작 (예: 첫 중간 배포 = `v0.1.0`).
- **MINOR** (+0.1.0): 기능 추가/일반 배포.
- **PATCH** (+0.0.1): 핫픽스.
- **MAJOR**: 호환성 깨지는 큰 변경 (정식 출시 시 `v1.0.0`).
- 태그는 반드시 **annotated (`-a`)** — 작성자/날짜/메시지가 남는다. **lightweight 금지.**

## 5. 핫픽스 정책 — 기본 "앞으로만 진행", 다중 유지 시 전환

- 중간 배포는 임시본. 버그가 나면 트렁크에서 고치고 다음 태그로 배포한다.
- 유지(maintenance) 브랜치를 **미리** 만들지 않는다.
- **예외 1 (단발)**: 현장에 특정 배포 버전이 살아남아 그 버전만 독립 패치해야 하는 상황이 실제로 오면, 그때 해당 태그에서 `hotfix/*` 가지를 잠깐 쳐서 패치 → 패치 태그(`vX.Y.Z+1`) → 트렁크로 머지백. 미리 만들지 말고 필요할 때만.
- **예외 2 (지속) — 크로스오버 주의**: 펌웨어는 웹과 달리, 고객/양산에 나간 순간 그 버전만 독립 패치해야 할 확률이 높다. 여러 SKU/고객이 서로 다른 버전을 동시에 들면, 임시 `hotfix/*`보다 `release/x.y` **유지 브랜치 모델**(§9)로 넘어가는 것이 정답이다. "예외"가 "일상"이 되는 시점을 망설이지 말고 전환하라.

## 6. 브랜치 위생

- 정본은 항상 `main`. `origin/HEAD`가 main을 가리키게 유지(GitHub 기본·CI 관습).
- 병합 끝난 작업 브랜치는 로컬·원격 모두 삭제. 잔재 브랜치를 쌓지 않는다.
- 삭제 전 반드시 `git rev-list --count main..<branch>`로 트렁크에 없는 고유 커밋이 0인지 확인. 0이 아니면 버릴지/머지할지/아카이브 태그로 보존할지 판단.
- 버리긴 아까운 실험본(예: SDK 포팅 시도)은 `archive/<주제>` 태그로 tip을 박아 보존한 뒤 브랜치는 삭제 → 히스토리는 남고 브랜치 목록은 깔끔.

## 7. 릴리스 추적성 (펌웨어 필수)

- **이미지에 git hash 임베드**: 현장 보드가 자기 버전(`git describe` 결과)을 보고할 수 있어야 태그 ↔ 펌웨어 매핑이 닫힌다.
- **바이너리 보관**: 태그는 `.bin`/`.hex`/`.elf`를 담지 못한다. CI 아티팩트나 릴리스 레지스트리에 보관하고, MCU별 `(커밋, 바이너리 해시)`를 **매니페스트**로 같이 박는다.
- **코드 서명**: secure boot를 쓰면 릴리스 태그 = 서명된 이미지. 릴리스 절차에 코드 서명 단계를 명시한다.

## 8. 실무 주의

- 태그는 커밋된 상태에만 박힌다 — 배포 태그는 **워킹트리가 깨끗하고 push된 tip**에서.
- git 정리(브랜치 삭제·fast-forward·태그)는 되돌리기 까다로우니, 실행 전 명령 목록을 검토하고 push 전에 로컬에서 먼저 확인한다.

## 9. 참고 — 규모가 커질 때의 대안 모델

본 표준(TBD + 태그)은 **소규모 단일 진화 제품의 기본값**이다. 상황이 바뀌면 아래로 전환을 검토한다. 선제 도입은 금물.

| 모델 | 언제 |
|------|------|
| **GitHub Flow** (모든 변경 = 짧은 브랜치 + PR) | 지금 표준에 리뷰/CI 게이트를 더 명확히 하고 싶을 때. 거의 동일하며 추천 변형. |
| **`release/x.y` 유지 브랜치** | 필드에 여러 버전이 동시에 살아 독립 패치가 필요해질 때. forward-only의 다음 단계. |
| **Git Flow** | 동시 유지 버전이 다수일 때. 단일 제품엔 과하지만 다중 유지엔 맞는 도구. |
| **매니페스트 / west 방식 (Zephyr)** | 멀티-MCU가 서로 다른 속도로 진화할 때. 매니페스트 저장소가 각 MCU 저장소의 SHA를 핀으로 박아 "한 세트 시스템 스냅샷"을 더 강하게 고정. MCU가 lockstep이면 모노레포+전체 태그로 충분. |
