## 2. ADC 트리거 전환 + HW 평균 필터 (A3.5, PR #6, 실보드 검증)

들어오는 ADC 값의 노이즈(널뛰기)를 줄이기 위해, 트리거 소스를 바꾸고 하드웨어 평균을 도입했다.

| 항목 | 변경 전 | 변경 후 |
|------|---------|---------|
| SOC 트리거 | RTI1 타이머 (1 kSPS) | **EPWM0_SOCA (85.032 kHz)** — PWM 마스터 클럭 |
| 노이즈 처리 | 없음 (raw) | **PPB 누적 HW 평균 N=64** (CPU 부하 0) |

- 6채널 전부 PWM 마스터(EPWM0)에 물려 샘플링이 스위칭 주파수(85 kHz)에 동기.
- 평균은 **PPB(Post-Processing Block) 하드웨어 누적** 방식 — CPU 개입 없이 노이즈가 √64 = 1/8로 감소.
- 평균 횟수 N은 매크로 **한 곳**(`ETA_ADC_OVERSAMPLE_LOG2`)으로 제어(6→64, 5→32 …), 코드 수정 후 재빌드만으로 조정.
- **실보드 HW 노이즈 측정으로 N=64 확정.** 6채널 0/3.3 V 추종·ISR 레이트 실측 OK.

#### 🖼 그림 2-A — "85 kHz 트리거로 어떻게 평균을 내나" (PPB 누적 평균)

> **그림 명세(skill용)**: 아래 ASCII를 **가로 타임라인 블록다이어그램**으로. 위 트랙 = 85 kHz 트리거 펄스 열(64개, 등간격 11.76 µs), 그 아래 = 트리거마다 변환 1회(s₁…s₆₄), 그 아래 = PPB 누산기 박스(PSUM += sᵢ / PCOUNT++), 오른쪽 끝에서 "PCOUNT==64 → SUM=PSUM>>6 → OSINT 1회"로 수렴. 하단에 창 폭·출력 레이트·노이즈·지연을 캡션 밴드로.

핵심: **트리거 1개당 변환 1회**. PPB(하드웨어 누산기)가 64회를 모아 **한 번만** 평균을 뱉는다 — CPU 개입 0.

```
                    ┌──────── 11.76 µs (85.032 kHz 트리거 간격) ────────┐
EPWM0_SOCA 트리거 :  ▲      ▲      ▲      ▲    ·····    ▲                  (트리거 64개)
변환(트리거당 1회):  s₁     s₂     s₃     s₄    ·····   s₆₄
                      \      |      |      /              /
PPB 하드웨어 누산기 :    PSUM += sᵢ ,  PCOUNT++            (CPU 개입 0, √ 자동)
                                       │
                          PCOUNT == 64 도달
                                       ▼
                      SUM = PSUM >> 6  =  (s₁+ … +s₆₄) / 64   ← 0~4095 스케일 유지
                                       ▼
                               OSINT 1회 발생 → ISR
   └──────────────── 평균 창 = 64 × 11.76 µs ≈ 753 µs ────────────────┘
   출력 레이트 = 85.032 kHz ÷ 64 = 1.33 kHz  │  노이즈 √64 = ÷8  │  그룹지연 ≈ 창/2 ≈ 370 µs
```

- **평균 = 우측 시프트**라서 N이 2의 거듭제곱(=64)일 때만 하드웨어가 나눗셈 없이 자동 평균한다(`SUM = PSUM >> 6`). 상세 [[am263p_adc_ppb_averaging]] §1.
- **ISR이 EOC가 아니라 "평균 완료(OSINT)"에 뜬다** → ISR 레이트 = 85.032 kHz ÷ 64 ≈ **1.33 kHz**.

#### 🖼 그림 2-B — "그 상태에서 우리가 보고 있는 데이터는 무엇인가"

> **그림 명세(skill용)**: **3단 레이어 스택**. ①원신호(노이즈 큰 톱니, 흐리게/점선) → ②PPB 평균 출력(매끈한 선, √64=÷8, 살짝 오른쪽으로 밀린 ≈370 µs 지연 표시) → ③GUI 표시(②위에 100 ms 간격 점 스냅샷). "우리가 보는 값 = ②를 ③으로 들여다본 것" 화살표.

우리가 GUI에서 보는 값은 raw ADC가 아니라 **최근 64변환의 하드웨어 평균**이고, 그것을 UART 텔레메트리 10 Hz로 스냅샷한 것이다.

```
① ADC raw (개념상 트리거마다, 85 kHz)   ▁▇▂▆▁▅▇▂▆▁▇▃   ← 노이즈 큰 원신호(직접 안 봄)
② PPB 평균 출력 (칩 내부 1.33 kHz)      ▄▄▄▅▅▄▄▄▄▄▄▄   ← √64=÷8 매끈, 단 ~370 µs 지연
③ GUI 표시 (UART 텔레메트리 10 Hz)      •         •     ← ②를 100 ms마다 1점씩 스냅샷
                                        └ 우리가 보는 데이터 = ②(64변환 HW 평균)의 10 Hz 스냅샷
```

- **대가**: 노이즈는 1/8로 줄지만 **출력이 1.33 kHz로 느려지고 그룹지연 ~370 µs**가 붙는다 → 제어루프/FOD엔 부담. 이 한계가 §3 리피터 버스트 전환의 동기다.

## 3. ADC 필터 방식 전환: 누적 평균 → 리피터 버스트 블록평균 (A5, PR #11, 실보드 + Saleae 검증)

§2의 PPB 누적 평균(N=64)은 노이즈는 잘 줄지만, 트리거당 여러 주기에 걸쳐 값이 나와 **실효 출력 레이트가 1.33 kHz로 떨어지는** 한계가 있었다. 이를 **한 트리거 안에서 16회 연속 변환 → 블록평균**하는 "리피터 버스트" 방식으로 전환했다.

- **효과**: 평균(노이즈 억제)은 유지하면서 **출력 샘플레이트가 다시 85 kHz/인스턴스로 복귀**(N과 무관).
- main 머지 완료(**PR #11**, CI 통과), 실보드에서 6채널 라이브 + 스코프 OSINT 85 kHz 검증.
- 현재 운영값 **N=16** (최종 N은 아래 §4 FFT 측정 후 튜닝).

#### 🖼 그림 3-A — "누적 평균은 왜 1.33 kHz로 떨어지나"

> **그림 명세(skill용)**: §2 그림 2-A와 같은 축을 재사용하되 **"64 트리거가 모여야 출력 1개"**를 강조. 트리거 64칸을 한 묶음으로 브래킷 치고, 그 오른쪽에 출력 1점. "85.032 kHz ÷ 64 = 1.33 kHz" 나눗셈을 큰 글씨로.

누적 평균은 한 샘플씩 64개 트리거에 **분산**해서 모은다 → 출력 1개를 얻는 데 64주기(≈753 µs)가 필요 → 실효 출력이 `85.032 kHz ÷ 64 = 1.33 kHz`로 떨어진다. (원리 도해는 §2 그림 2-A.)

#### 🖼 그림 3-B — 리피터 버스트 블록평균 원리 (한 트리거 안에서 16회)

> **그림 명세(skill용)**: **한 주기(11.76 µs) 확대 뷰**. 왼쪽에 트리거 1펄스 → "리피터(오버샘플링, repCount+1=16)" 박스 → 16개 변환이 **한 덩어리로 뭉쳐** 백투백(3.12 µs 폭, 주기의 26.5%) → PPB 블록평균 → OSINT 1회. 주기 나머지 73.5%는 idle(빈 막대). "출력 85 kHz (N 무관)" 강조.

핵심 차이: 16회 변환이 **여러 트리거에 흩어지지 않고 한 트리거 안에** 몰려 일어난다. 그래서 **출력은 매 트리거(=85 kHz) 그대로**, 평균만 얹힌다.

```
                     ┌──────────── 11.76 µs 주기 ────────────┐
EPWM0_SOCA 트리거 :  ▲                                        ▲   (주기당 1개)
                     │
              리피터(오버샘플링, repCount+1 = 16)
                     ▼   한 트리거를 16 펄스로 복제, 백투백 발사
변환(한 주기 안):  [c₁c₂c₃c₄c₅c₆c₇c₈c₉c₁₀c₁₁c₁₂c₁₃c₁₄c₁₅c₁₆]   ← 3.12 µs 안에 16회
                     └───────────────┬───────────────┘
PPB 블록평균 :          PSUM += cᵢ (16회) , PCOUNT == 16
                        SUM = PSUM >> 4  =  (c₁+ … +c₁₆) / 16
                                     ▼
                              OSINT 1회 → ISR       ← 매 트리거마다!
   주기 점유 :  [■■■■■■■■■■■■■■■■]················ idle ················
                버스트 3.12 µs (26.5%)                나머지 73.5% 유휴
   출력 레이트 = 85.032 kHz (N과 무관, 트리거 레이트 그대로)  │  그룹지연 ≈ 버스트/2 ≈ 1.5 µs
```

메커니즘·SDK 정본 [[am263p_adc_repeater_burst]].

#### 🖼 그림 3-C — 누적 vs 버스트 나란히 (같은 N개 변환, 다른 시간축)

> **그림 명세(skill용)**: **좌우 2열 대비 다이어그램**. 왼쪽=누적(N=64, 트리거 64개에 s 하나씩 분산, 출력 1.33 kHz), 오른쪽=버스트(N=16, 트리거마다 16연발, 출력 85 kHz). 같은 "N회 평균"인데 **시간축에 흩느냐 한 트리거에 몰느냐**가 유일한 차이임을 시각화.

```
        누적 PPB (A3.5, N=64)                리피터 버스트 (A5, N=16)  ← 현재
트리거   ▲  ▲  ▲  ▲  ·····  ▲ (64주기)        ▲            ▲            ▲
변환     s  s  s  s  ·····  s                 [16연발]     [16연발]     [16연발]
         └──────── 모아서 ────────┘            ▼            ▼            ▼
출력                  ▼ 1회                    매 트리거 1회
레이트    85k / 64 = 1.33 kHz  ✗느림           85 kHz 유지  ✓
평균 창   트리거 주기에 걸친 저속(≈753 µs)      한 트리거 순간(3.12 µs)
억제 대상 저주파 드리프트                        순간 백색·양자화 잡음
변환예산  N과 무관                              N × cadence ≤ 주기 (N ≤ ~41)
```

> **⚠️ 1.33 kHz vs 5.3 kHz 혼동 주의(그림 캡션에 명시)**: `1.33 kHz`는 **N=64 누적**(=85k/64), 아래 Saleae 절의 `5.3 kHz`는 **"현재 N=16을 만약 누적으로 돌렸다면"**(=85k/16)의 가정치다. 서로 다른 N을 가리킨다. 버스트로 전환한 현재 실측 출력은 **85.03 kHz**.

#### 🖼 그림 3-D — 현재 설정 (하나의 손잡이 N)

> **그림 명세(skill용)**: 작은 **설정 카드/표**. `ETA_ADC_OVERSAMPLE_LOG2` 한 매크로가 리피터 repCount·PPB LIMIT·SHIFT 세 곳으로 갈라져 나가는 fan-out 화살표.

| 항목 | 현재 값 | 위치 / 근거 |
|------|---------|-------------|
| N 손잡이(단일 소스) | `ETA_ADC_OVERSAMPLE_LOG2 = 4U` → N = 2⁴ = **16** | `src/bsp/eta_bsp_adc.h:31` (`_Static_assert ≤ 10`) |
| 리피터 반복 횟수 | `repCount = (1<<4) − 1 = 15` → 16 펄스 | `eta_bsp_adc.c` 런타임 재기입 |
| PPB 배치 크기 | `setPPBCountLimit(16)` | 〃 |
| PPB 평균 시프트 | `setPPBShiftValue(4)` (÷16) | 〃 |
| N 상한(변환예산) | ~41 (cadence 285 ns × N ≤ 11.76 µs) | [[am263p_adc_instance_allocation]] |

매크로 한 줄(4→5→6 …)만 바꿔 재빌드-flash하면 리피터·PPB 세 값이 동반 추종. 최종 N은 §4 FFT 측정 후 확정.

### Saleae 로직 분석기 실측 (6/30~7/1)

버스트 방식이 실제로 "한 트리거 = 한 번 완결"로 동작하는지 로직 분석기로 실증했다.

| 측정 항목 | 결과 |
|-----------|------|
| 트리거당 ISR 발생 횟수 | **정확히 1회** (12,260 / 12,260 트리거, 누락·중복 0) → 버스트 실증 |
| 유효 샘플레이트 | **85.03 kHz** (누적 방식이었다면 ≈5.3 kHz여야 함) |
| 트리거 → 버스트 종료 | 3.12 µs (주기의 26.5%) |
| ISR 처리 시간 | 0.304 µs |

> 만약 종전 누적(N=64) 방식이었다면 ISR이 트리거마다 나오지 않았을 것 — 트리거당 정확히 1회 ISR이 곧 버스트(블록평균)가 맞다는 직접 증거.

#### 🖼 그림 3-E — 원본 CSV 실측 파형 (기대대로 동작 실증)

> **그림 명세(skill용)**: **디지털 로직 애널라이저 스타일 2채널 타이밍도**. 데이터 소스 = `raw/adc_repeater_burst_timing/digital.csv`(ch0=EPWM2_A, ch1=GPIO95). 첫 3주기(0~33 µs) 창을 그린다. 위 트랙 EPWM2_A(듀티 46.6% 구형파), 아래 트랙 GPIO95(주기마다 폭 0.304 µs 좁은 펄스 1개). "주기당 GPIO95 펄스 정확히 1개"가 한눈에 보이게. 아래 표의 실제 엣지 시각(µs)을 눈금으로.

원본 CSV에서 **실제로 뽑은 첫 3주기**의 엣지 시각(전량 12,260주기 중 대표 창):

| 주기 | EPWM2_A ↑ (µs) | GPIO95 ↑ (µs) | GPIO95 ↓ (µs) | ISR 폭 (µs) | 직전 주기와 간격 |
|------|----------------|----------------|----------------|-------------|------------------|
| 1 | 3.446 | 3.460 | 3.764 | 0.304 | — |
| 2 | 15.206 | 15.206 | 15.508 | 0.302 | 11.760 µs |
| 3 | 26.966 | 26.996 | 27.300 | 0.304 | 11.760 µs |

→ 주기 간격 **11.760 µs 일정 = 85.032 kHz**, GPIO95 펄스 **주기당 정확히 1개**, 폭 **0.304 µs**. 설정(§3-B/3-D)대로 "한 트리거 = 한 번 완결"이 실물에서 그대로 나온다.

```
 t(µs) │  3.4      8.9     15.2     20.7     27.0     32.4
EPWM2_A│  ┌─────────┐        ┌─────────┐        ┌─────────┐
 (HS1) │──┘         └────────┘         └────────┘         └────   T=11.76 µs, 듀티 46.6%
       │
GPIO95 │  ╻                  ╻                  ╻
(OSINT)│──╹──────────────────╹──────────────────╹────────────────  주기당 1펄스, 폭 0.304 µs
       │  ↑ EPWM2_A↑와 사실상 동시(+14~30 ns)
```

**증명 논리 (그림 캡션)**: 버스트라면 트리거(=85 kHz)마다 OSINT 1회 → 위 파형처럼 주기당 펄스 1개. 만약 누적(N=16)이었다면 16주기에 1회(≈5.3 kHz)만 떴을 것. 실측은 **12,260 트리거 / 12,260 펄스 = 1:1, 85.03 kHz** → 버스트 확정.

#### 🖼 그림 3-F — 한 주기 내부 재구성 (트리거 → 버스트 → ISR)

> **그림 명세(skill용)**: **한 주기(11.76 µs) 확대 밴드**. EPWM0 ZERO(트리거, 위상 역산 위치)에서 시작 → 3.12 µs 구간에 "16변환 버스트 + OSINT→IRQ 진입지연" → GPIO95↑ → 0.304 µs ISR 실행 → GPIO95↓ → 나머지 idle. 하단에 "cadence ≤ 3.12/16 = 195 ns/변환(상한)".

EPWM0 ZERO(트리거)는 직접 캡처 안 했고 EPWM2_A 위상에서 역산(트리거 = EPWM2_A↑ − 0.265·T, 두 방법 24 ns 내 일치).

```
 EPWM0 ZERO                    GPIO95↑          GPIO95↓
 (트리거, 위상역산)             (ISR 진입)        (ISR 종료)
      │◄──────── 3.12 µs ────────►│◄─ 0.304 µs ─►│
      │  16변환 백투백 버스트       │   ISR 실행    │◄──────── idle ~8.3 µs ────────►│
      │  + OSINT→IRQ 진입지연       │              │
      └───────────────────────── 11.76 µs 주기 (26.5% 점유) ─────────────────────────┘
   실효 cadence ≤ 3.12 µs / 16 = 195 ns/변환  (상한 — IRQ 진입지연 포함; 정적 모델 285 ns보다 빠름)
```

실측 리포트 정본 [[adc_repeater_burst_timing]], 원본 데이터 `raw/adc_repeater_burst_timing/digital.csv`(48,724 엣지, ≈144 ms).

### 변환시간 예산 정밀 산정 (리피터 N 상한)

버스트 N을 얼마까지 키울 수 있는지 산정했다. 1회 변환 cadence ≈ **285 ns**(S+H 80 ns + EOC 205 ns), 85 kHz 주기(11.76 µs)당 **~41 변환** 수용 가능 → **리피터 N 상한 ~41**. 즉 N=16은 예산 안에 충분히 여유가 있어, FFT 결과에 따라 상향 여지가 있다.

### SW 이동평균(A6) — 왜 도입하고, 링버퍼로 어떻게 구현하나 [착수 전]

#### 🖼 그림 3-G — HW 버스트 ⟂ SW 이동평균 (직교, 다른 시간축)

> **그림 명세(skill용)**: **두 시간축 대비**. 왼쪽=버스트가 다루는 "한 트리거 안 3.12 µs 좁은 창"(줌인), 오른쪽=이동평균이 다루는 "M개 트리거에 걸친 넓은 창 M×11.76 µs"(줌아웃). 둘이 겹치지 않는(=직교) 서로 다른 스케일임을 강조. "함께 쓰면 순간잡음 + 시간축잡음 동시 억제".

**도입 이유**: 리피터 버스트(HW)는 **한 트리거 순간(3.12 µs)의 좁은 창** 안 잡음(순간 백색·양자화)만 줄인다. 하지만 **트리거 주기에 걸친** 저주파 드리프트·간헐 스파이크는 손대지 못한다. SW 이동평균은 **M개의 서로 다른 트리거 샘플**을 시간축으로 평균하는 저역통과라 버스트와 **경쟁이 아니라 직교** — 같이 쓰면 두 축의 잡음을 동시에 억제한다.

| 축 | HW 리피터 버스트 | SW 이동평균 |
|----|------------------|-------------|
| 동작 창 | 한 트리거 내 3.12 µs | M 트리거 × 11.76 µs |
| 억제 대상 | 순간 백색·양자화 | 시간축 저역·드리프트·스파이크 |
| 출력 레이트 | 85 kHz | 85 kHz (매 샘플 갱신) |
| 그룹지연 | ≈ 버스트/2 ≈ 1.5 µs | (M−1)/2 × 11.76 µs |
| 비용 제약 | 변환시간 예산 (N ≤ ~41) | CPU·RAM(M칸)·허용지연 |

#### 🖼 그림 3-H — 링버퍼 이동평균 구현 (러닝합, 나눗셈·루프 없음)

> **그림 명세(skill용)**: **원형 버퍼(링) 그림** M칸(예: 8). idx 포인터가 "가장 오래된 칸 = 다음 쓸 칸"을 가리키고, 새 샘플이 그 칸을 덮어쓰는 화살표. 옆에 러닝합 갱신 3줄(빼고-더하고-나누기)을 코드 박스로. "매 트리거 1회, O(1)".

매 트리거 새 샘플이 링버퍼에서 가장 오래된 값을 덮어쓰고, **러닝합**만 갱신하므로 트리거당 O(1)(루프·나눗셈 없음).

```
 링버퍼 buf[M], 인덱스 idx, 러닝합 S     (예: M = 8 = 2³)
 ┌────┬────┬────┬────┬────┬────┬────┬────┐
 │ x₀ │ x₁ │ x₂ │ x₃ │ x₄ │ x₅ │ x₆ │ x₇ │  ← 매 트리거 새 샘플이 가장 오래된 칸을 덮음
 └────┴────┴────┴────┴────┴────┴────┴────┘
             ▲ idx (다음 쓸 자리 = 지금 가장 오래된 값)

 매 샘플(ISR write):
   S       += new − buf[idx];     // 러닝합: 나갈 값 빼고 새 값 더함 (루프 없음)
   buf[idx] = new;
   idx      = (idx + 1) & (M − 1); // M이 2의 거듭제곱이면 & 마스크로 wrap
   out      = S / M;               // M = 2ⁿ이면 >> n (나눗셈도 시프트로)

 출력 = 매 트리거 갱신(85 kHz 유지)  │  그룹지연 = (M−1)/2 × 11.76 µs
```

- **배치**: A6 — `eta_alg_filter`(ALG 레이어) 신설, **ISR write 방식 먼저** 구현해 DMA(EDMA) 오프로드 검증과 분리(권장순). CPU 부하 실측(R5F @400 MHz, ~784 사이클/ISR [추정]) + 윈도우 M 튜닝 예정. 정본 [[am263p_adc_repeater_burst]] §5, 로드맵 [[adc]] §A6.

## 4. ADC 노이즈 최소화 측정 계획 수립 (6/30)

리피터 버스트 전환(§3)으로 **85 kHz 샘플레이트는 되찾았으니**, 남은 과제는 **노이즈 최소화**다. 스코프 FFT로 노이즈 성격을 판별해 **최종 N / 트리거 위상**을 정하는 측정 계획을 수립했다.

- **판별 로직**: 노이즈가 백색(랜덤)이면 → 평균 N을 키우면 개선. 스위칭 주파수(85 kHz) 고조파와 상관되면 → ADC 트리거가 EPWM에 위상 고정돼 있어 평균이 무효 → **트리거 위상**이 유일한 레버.
- **프로브 포인트**: **I_COIL_SEN(J3.28)** — 코일전류 CT를 정류·평활한 DC 포락선이라 정상상태에선 공칭 DC → FFT에 뜨는 AC = 노이즈로 해석된다. (절대 노이즈 플로어를 정밀 측정할 땐 GA_Vin·GA_Iin이 더 깨끗하나, FOD 목적엔 I_COIL_SEN이 직접적.)
- **장비·설정**: MSOX3104T, Hanning·0~500 kHz·RMS 평균.

## 다음 작업 — I_COIL_SEN 노이즈 측정 → ADC 안정화 → FOD 관찰

이번 주 착수한 FOD 관찰(§5)을 정식 절차로 수행한다. **이물(금속편)이 코일 결합·부하를 바꾸면 코일전류가 변하고, 그것이 I_COIL_SEN의 DC 레벨 시프트로 나타난다**는 가설을, **먼저 센싱을 안정화한 뒤** 정상 vs 이물을 비교해 검증한다.

- **신호체인**: I_COIL_SEN(J3.28)은 코일전류 CT(T1, 7 mV/A) → 쇼트키 정류 → RC 평활을 거친 **DC 포락선**. 이물에 의한 레벨 변화 관측에 적합.
- **실행 순서 (안정화 선행)**:
  1. **정상에서 노이즈 측정** — 전력단 스위칭 중, I_COIL_SEN에 스코프(AC·FFT) + GUI 10 Hz CSV로 노이즈 성격(백색 vs 85 kHz 스위칭 상관) 판별 + 표준편차 σ 산출.
  2. **N값 / 트리거 위상 튜닝 → ADC 안정화** — 노이즈가 백색이면 평균 N 상향, 스위칭 상관이면 트리거 위상 조절(위상 고정 샘플링이라 상관 노이즈엔 평균이 무효).
  3. **안정화 후 FOD(이물) 관찰** — 낮춘 σ 위에서 이물 투입(OC 임계 이하) → DC 레벨 시프트 Δ 측정 → **검출성 Δ/σ** 평가.
- σ를 먼저 낮춰두면 Δ/σ가 유리하고, 안정화 튜닝을 이물 변수 없이 깨끗하게 할 수 있다.

### 후속 (FOD 이후)

1. **SW 이동평균 필터(A6)** — HW 버스트 평균 위에 SW ring buffer 이동평균을 얹어 스파이크+노이즈 동시 억제.
2. **ADC A4 실보드 교차검증** — 멀티미터 기준값으로 스케일링 정확도 확인.
