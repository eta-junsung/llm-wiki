| 비유 | 실제 |

## 1. 이 작업이 뭔가

**목표**: BP-CC3351을 LP-AM263P에서 구동 — WiFi scan/connect + BLE HCI 실동작.

**왜 이게 일인가**: TI는 이 WiFi 카드를 **LP-AM243(구형 보드)** 용으로만 공식 지원함. LP-AM263P용 예제·드라이버 설정은 제공하지 않음.
→ AM243용 SDK 예제를 AM263P에 **직접 이식(포팅)**해야 함. 비공식 작업.

> **한 줄 비유**: AM243(기존 PC)에서 돌던 OS+WiFi 드라이버를 AM263P(새 PC)에 설치하는 것. 새 PC 전용 설치 가이드는 없음.

## 3. 가장 길게 막혔던 곳 — S3 (27라운드, 약 1개월+)

### 무슨 상황이었나

단계 1~2는 됐음:
```
BIOS(SBL)가 USB(플래시) 읽기    → 성공 ✓
OS(App)를 RAM에 올리기          → 성공 ✓
RAM에서 뜬 OS가 USB를 다시 인식 → 실패 ✗  ← 여기
```

**"USB에 부팅은 됐는데, 부팅된 OS가 방금 자기를 띄운 그 USB를 다시 못 잡는 상황."**

### 왜 1·2는 됐는데 3만 안 됐나

BIOS와 OS 드라이버가 **각자 독립적으로 USB 컨트롤러를 초기화**하기 때문.

- BIOS(SBL): TI가 만든 것, 핀 설정이 원래 맞음 → USB 잘 읽음
- OS 드라이버(App): 자체 설정 파일(example.syscfg)로 재초기화 → **이 파일에 핀이 잘못 박혀 있었음**

물리적으로 잘못 꽂힌 게 아니라, 소프트웨어 설정 파일의 핀 번호가 틀렸던 것.

### 왜 이게 어려웠나

USB 종류 자체가 달라서 뭐가 문제인지 파악이 어려웠음:
- AM243(구형): Quad-SPI 방식 플래시 (일반 규격)
- AM263P(신형): Octal xSPI 8D DDR 방식 플래시 (고속 규격, 핀 수·프로토콜·초기화 시퀀스 전부 다름)

드라이버 코드·타이밍·프로토콜을 27라운드 동안 수정했으나 계속 실패.

### 어떻게 해결했나 (R28)

핀 설정 도구(SysConfig)에서 자동 추천(`$suggestSolution`)으로 잡힌 핀 7개가 전부 엉뚱한 곳에 배치돼 있었음.
→ **수동 고정(`$assign`)으로 교정** — 설정 파일 한 줄 수정으로 즉시 해결.

> **비유**: 27라운드 동안 USB 드라이버 코드를 뜯어봤는데, 실제 원인은 설정 파일에 적힌 핀 번호가 틀렸던 것. 번호 고치니 바로 인식.

## 4. 지금 어디 — S5/S6 경계

- **S5**: WiFi 칩(CC33xx) 펌웨어 로드 → `Hardware init DONE!` 출력 확인. 단, 칩과 통신 핸드셰이크는 미완.
- **S6 막힘**: SPI 통신 링크 (`SPI not responsive`) — PC ↔ WiFi 카드 사이 데이터 선이 응답 없음.

**다음 작업 (R35)**: WiFi 카드의 reset 신호(WLAN_EN) 해제 타이밍·극성 확인. 카드가 제대로 켜지고 있는지부터 검증.
