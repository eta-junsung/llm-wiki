# AM263P ADC 인스턴스 배치 — 표준·권장 방식

> **AM263P 플랫폼 설계 규칙 정본.** 신호를 어느 ADC 인스턴스/채널에 배정할지 결정하는 기준. 형제 정본: [[am263p_adc_rti_trigger]](트리거 결선·EOC ISR). 8kw 적용: [[adc_pinmap]].

## 전제 교정 (중요)

**여러 ADC 인스턴스를 쓰는 게 덜 안정적인 게 아니다.** AM263P 같은 모터·전력제어용 MCU가 ADC0~ADC4를 둔 이유 자체가 "동시에 여러 신호를 잡으라"는 것 — 멀티 인스턴스는 정상·표준 사용법이다. 진짜 차이는 **안정성이 아니라 ① 타이밍(동시성) ② 펌웨어 복잡도** 두 축이다.

## 핵심 하드웨어 사실

- **한 인스턴스 안의 여러 SOC = 직렬.** 인스턴스마다 변환기(SAR)·S/H가 **하나뿐**이라, 같은 인스턴스에 채널을 여러 개 걸면 **트리거가 같아도 순차 변환**된다. → 채널 간 **샘플 스큐**(뒤 채널은 앞 채널 변환시간만큼 늦게 샘플) 발생, 그리고 **모든 SOC 변환시간 합이 트리거 주기 안에 들어와야** 한다.
- **서로 다른 인스턴스 = 병렬.** 독립 하드웨어라 같은 트리거(예 RTI1)에 걸면 **물리적으로 동시 샘플**된다. 스큐 없음.

## 권장 결정 규칙 (안정성 아님 → 신호 특성 기준)

| 신호 성격 | 권장 배치 | 이유 |
|-----------|-----------|------|
| **동시성 중요 / 빠른 상관 신호** — 코일전류·입력전류, V·I 쌍 등 제어루프에 들어가는 것 | **서로 다른 인스턴스 + 공통 트리거** | 동시 샘플 → 위상·전력 계산 정확. 멀티 ADC의 존재 이유 |
| **느린 / 무상관 신호** — 온도, 느린 전압 | **한 인스턴스에 SOC로 몰기, 마지막 EOC에 인터럽트 1개** | 스큐 무의미, 펌웨어 단순(ISR 1개·flag 1개) |

## 펌웨어 복잡도 축

인스턴스를 하나 늘릴 때마다 **+ISR 1개, +int_xbar 라우팅 1개, +ready 플래그 1개**가 든다(8kw ADC0 추가가 정확히 이 비용). 그래서 **"동시성이 필요 없는 신호는 한 인스턴스에 먼저 채우는 것"**이 펌웨어를 단순하게 유지하는 타당한 **tiebreaker**다 — 단 **복잡도 절감이지 안정성은 아니다.**

## 진짜 불안정해지는 경우 (유일한 실제 instability)

한 인스턴스에 너무 많이 몰아서 **SOC 변환시간 합 > 트리거 주기**가 되면 → 다음 트리거 전에 시퀀스를 못 끝내 결과가 **밀리거나 덮어씌워진다.** 이게 유일한 실제 instability이고, **멀티 인스턴스로 분산하면 오히려 해소**된다.

## 한 줄 요약 (다음 보드 설계 시)

상관·고속 신호는 인스턴스를 나눠 **병렬·동시샘플**, 무상관·저속 신호는 한 인스턴스에 모아 **ISR 절약**. "인스턴스 채우는 순서"는 안정성이 아니라 **동시성 요구 + 인스턴스당 변환시간 예산**으로 결정. ★ 단, 실보드에선 **PCB 라우팅이 신호→인스턴스/채널을 거의 고정**하므로, **설계 단계에서** 위 기준으로 핀을 배정해두는 게 핵심.

## 8kw 적용 메모

현재 [[adc_pinmap]] 배치는 PCB 라우팅으로 고정된 상태:

- 상관 신호(GA_Vin·GA_Iin_SEN·I_LCC_SEN·I_COIL_SEN)는 가급적 서로 다른 인스턴스에 분산되는 것이 위 규칙에 부합 — 동시 샘플로 위상·전력 계산 정확도 확보.
- 무상관·저속(Temp_Module1/2)은 한 인스턴스에 몰아도 무방.
- ⚠️ 현 배치에서 **ADC1 = Temp_Module2(AIN0) + GA_Iin_SEN(AIN1)** 동거: 입력 전류(상관 신호)가 온도(저속)와 같은 인스턴스라 직렬 변환·스큐 발생. **현 보드는 라우팅 고정이라 그대로 운용**하되, 다음 보드 설계 시 입력 전류는 입력 전압(GA_Vin)과 동시 샘플 가능하도록 별도 인스턴스 배정 검토.
